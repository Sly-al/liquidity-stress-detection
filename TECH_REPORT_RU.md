# RU Liquidity Sentinel: техническое описание

## 1. Описание компании и контекста

ПСБ рассматривается как крупный системно значимый банк, для которого управление рублевой ликвидностью является ежедневной задачей казначейства. Цель системы — собрать разрозненные публичные сигналы денежного рынка в единый наблюдаемый контур, чтобы казначей видел не только отдельные показатели, но и агрегированную оценку стресса.

В прототипе реализована система `RU Liquidity Sentinel`: расчетный backend на Python и интерактивный dashboard на Streamlit. Система строит индекс `Liquidity Stress Index` (`LSI`) от 0 до 100, показывает статус рынка и объясняет вклад каждого модуля.

## 2. Бизнес-проблема

Стресс ликвидности может развиваться быстро, а ранние признаки распределены между разными источниками: ЦБ РФ, Минфин, ФНС, Росказна и другими публичными площадками. Ручной мониторинг таких данных занимает время, плохо масштабируется и усложняет сравнение текущей ситуации с историческими эпизодами.

Решение автоматизирует три слоя работы:

- сбор и подготовку данных;
- расчет независимых сигналов по пяти экономическим каналам;
- агрегацию сигналов в понятный индекс с визуальным и текстовым объяснением.

## 3. Задача и реализованное решение

В проекте реализованы пять расчетных модулей:

- `М1` — усреднение обязательных резервов и RUONIA;
- `М2` — аукционы репо ЦБ;
- `М3` — размещение ОФЗ;
- `М4` — налоговый период и сезонность;
- `М5` — средства федерального казначейства.

Каждый модуль формирует набор количественных признаков и флагов. Количественные признаки нормализуются через rolling MAD за окно около трех лет. Затем признаки передаются в агрегационный слой, который строит `LSI`.

Реализованные пользовательские выходы:

- индекс `LSI` 0-100;
- статус `ЗЕЛЁНЫЙ`, `ЖЁЛТЫЙ`, `КРАСНЫЙ`;
- вклад каждого модуля;
- графики сигналов;
- backtest на стресс-эпизодах;
- sensitivity analysis;
- аналитический чат.

## 4. Ожидаемый результат и покрытие требований

Минимальные требования закрыты следующим образом:

- все пять модулей реализованы в `sentinel/modules.py`;
- MAD-нормализация реализована в `sentinel/utils.py`;
- агрегация в LSI реализована в `sentinel/aggregation.py`;
- интерпретация вклада модулей строится через коэффициенты линейной модели и вклад признаков;
- dashboard реализован в `app.py`;
- backtest реализован в `sentinel/backtest.py`;
- исторические эпизоды: декабрь 2014, февраль-март 2022, август 2023.

Бонусный слой реализован через вкладку `Аналитик`. В ней используется Hugging Face Inference Providers для живого LLM-ответа. Если внешний сервис недоступен, система возвращает локальный аналитический ответ, чтобы пользователь не видел техническую ошибку.

## 5. Общая архитектура системы

Pipeline устроен так:

```text
data.py
  ↓
modules.py: M1, M2, M3, M4, M5
  ↓
utils.py: rolling MAD normalization
  ↓
aggregation.py: RidgeCV + LSI + module contributions
  ↓
backtest.py / llm.py
  ↓
app.py: Streamlit dashboard
```

Главная точка сборки — `build_liquidity_sentinel()` в `sentinel/pipeline.py`. Она возвращает объект `SentinelResult`, где находятся исходные данные, результаты модулей, агрегация, backtest, sensitivity analysis, статус публичных источников и автоматический комментарий.

### 5.1 Данные и backend

В `sentinel/data.py` есть два типа логики.

Первый тип — download hooks для публичных источников из ТЗ. Они скачивают страницы и Excel-файлы в `data/cache/`. Это нужно для демонстрации производственного направления интеграции.

Второй тип — детерминированный калиброванный набор данных. Он нужен, чтобы проект стабильно запускался без зависимости от изменения форматов сайтов. В данных заложены экономические зависимости между модулями и исторические стресс-эпизоды:

- декабрь 2014;
- февраль-март 2022;
- август 2023.

Такой подход позволяет показать полный аналитический цикл и одновременно оставить точки подключения реальных парсеров.

### 5.2 М1 — усреднение обязательных резервов

Модуль считает:

- фактические среднедневные остатки на корсчетах;
- обязательные резервы;
- спред усреднения;
- RUONIA;
- флаг конца периода;
- флаг перебора нормы.

Сигналы:

- `m1_mad_spread`;
- `m1_mad_ruonia`;
- `m1_overbuffer_flag`.

Экономическая интерпретация: рост спреда резервов вместе с ростом RUONIA указывает на повышенный спрос банков на ликвидность.

### 5.3 М2 — аукционы репо ЦБ

Модуль считает:

- спрос на репо;
- объем размещения;
- `cover ratio`;
- ключевую ставку;
- ставку отсечения;
- спред ставки к ключевой.

Сигналы:

- `m2_mad_cover`;
- `m2_mad_rate_spread`;
- `m2_demand_flag`.

На неаукционные дни последнее известное значение аукциона переносится вперед, чтобы dashboard не показывал пустые значения между аукционами.

### 5.4 М3 — размещение ОФЗ

Модуль считает:

- объем предложения;
- спрос;
- размещение;
- `cover ratio`;
- средневзвешенную доходность;
- спред доходности к бенчмарку.

Сигналы:

- `m3_mad_cover`;
- `m3_mad_yield_spread`;
- `m3_nedospros_flag`;
- `m3_perespros_flag`.

Низкий `cover ratio` трактуется как стрессовый сигнал: инвесторы и банки менее готовы связывать ликвидность в ОФЗ.

### 5.5 М4 — налоговый период и сезонность

Модуль формирует календарные признаки:

- налоговая неделя;
- ключевая налоговая дата;
- конец месяца;
- конец квартала;
- `Seasonal_Factor`.

`Seasonal_Factor` используется как мультипликатор в агрегационном слое. Это снижает риск двойного счета: налоговый период не просто добавляется отдельной суммой, а контекстуализирует сигналы остальных модулей.

### 5.6 М5 — средства федерального казначейства

Модуль считает:

- остатки бюджетных средств в банках;
- депозиты ЕКС;
- количество банков-участников;
- недельный бюджетный отток;
- месячный отток депозитов;
- флаг бюджетного оттока.

Сигналы:

- `m5_mad_cbr`;
- `m5_mad_roskazna`;
- `m5_budget_drain_flag`.

Экономическая логика: отток бюджетных средств из банковской системы усиливает спрос на краткосрочное фондирование и может сопровождаться ростом ставок.

### 5.7 ML-агрегация и интерпретация

Для агрегации выбран `RidgeCV` из `scikit-learn`.

Причины выбора:

- модель устойчива на коррелирующих признаках;
- быстро обучается;
- не требует сложной инфраструктуры;
- коэффициенты легко объяснить бизнес-пользователю;
- вклад каждого модуля можно рассчитать явно.

Pipeline агрегации:

1. Берутся MAD-признаки и флаги всех модулей.
2. Признаки стандартизируются через `StandardScaler`.
3. `RidgeCV` подбирает регуляризацию по сетке `alpha`.
4. Модель прогнозирует стресс-оценку.
5. Результат умножается на `Seasonal_Factor`.
6. Значение ограничивается шкалой 0-100.
7. Рассчитываются вклады `M1-M5`.

Статусы:

- `0-40` — `ЗЕЛЁНЫЙ`;
- `40-70` — `ЖЁЛТЫЙ`;
- `70-100` — `КРАСНЫЙ`.

Интерпретация строится через сумму положительных вкладов стандартизированных признаков внутри каждого модуля. Это дает понятное объяснение: не только насколько высок LSI, но и какие каналы создали сигнал.

### 5.8 Backtest и sensitivity analysis

Backtest проверяет, поднимался ли LSI в известных стресс-эпизодах:

- декабрь 2014;
- февраль-март 2022;
- август 2023.

Для каждого эпизода считаются:

- средний LSI;
- максимальный LSI;
- средний LSI до эпизода;
- количество дней раннего предупреждения;
- факт детекции.

Sensitivity analysis показывает, как меняется итоговый LSI при изменении вклада каждого модуля на `±20%`. Это нужно для проверки устойчивости индекса к весам и модельным допущениям.

### 5.9 Dashboard и GUI

Dashboard реализован в `app.py` на Streamlit.

#### Вкладка `Дашборд`

Показывает:

- текущий LSI;
- статус рынка;
- RUONIA;
- `cover ratio` репо;
- `cover ratio` ОФЗ;
- график LSI за выбранный период;
- вклад модулей;
- автоматический комментарий;
- все текущие метрики по группам M1-M5;
- историю выбранных метрик в нормированной шкале.

Эта вкладка предназначена для ежедневного мониторинга.

#### Вкладка `Модули`

Пользователь выбирает модуль и видит:

- график двух ключевых сигналов;
- последние строки расчетной таблицы;
- текущие значения LSI и статуса.

Эта вкладка нужна для расшифровки, какой именно экономический канал дал сигнал.

#### Вкладка `Backtest`

Показывает:

- таблицу исторических стресс-эпизодов;
- sensitivity analysis;
- качество модели на train и holdout;
- коэффициенты ML-модели.

Эта вкладка нужна для проверки надежности и объяснимости индекса.

#### Вкладка `Аналитик`

Реализует чат на естественном языке. Пользователь может спросить:

- почему вырос LSI;
- что происходило в августе 2023;
- какие периоды были самыми стрессовыми;
- какие модули внесли основной вклад.

LLM получает только контекст системы: LSI, модульные сигналы, флаги, backtest и историю диалога. Если внешний LLM-сервис недоступен, включается локальный ответ без вывода технической ошибки пользователю.

#### Вкладка `Источники`

Показывает статус загрузки публичных источников и пути к кешированным файлам. Она нужна для контроля процесса обновления данных.

### 5.10 LLM-модуль

LLM-слой реализован в `sentinel/llm.py`.

Используется Hugging Face Inference Providers и модель:

```text
openai/gpt-oss-20b:cheapest
```

Причина выбора: это рабочий недорогой маршрут через Hugging Face router. Суффикс `:cheapest` просит выбрать наиболее экономичного провайдера для выбранной модели.

Prompt включает:

- системную роль аналитика казначейства;
- фиксированную шкалу LSI;
- последний snapshot показателей;
- summary по запрошенному периоду;
- топ стрессовых дней;
- backtest;
- историю диалога.

В prompt явно запрещено выдумывать значения вне переданного контекста и показывать пользователю технические имена полей.

### 5.11 Тестирование и запуск

Основные команды:

```bash
python3 -m pip install -r requirements.txt
python3 run_pipeline.py
python3 -m pytest -q
python3 -m streamlit run app.py --server.address 0.0.0.0 --server.port 8501
```

Тест `tests/test_pipeline.py` проверяет, что pipeline строится, LSI находится в диапазоне 0-100, есть вклады модулей и все три стресс-эпизода детектируются.

### 5.12 Ограничения прототипа

Система является аналитическим прототипом. Основное ограничение — использование калиброванного воспроизводимого набора данных вместо полноценного production-парсинга всех источников. Это сделано осознанно, чтобы демонстрация не ломалась из-за изменений HTML/Excel-форматов публичных сайтов.

Для production-версии нужно:

- заменить синтетический слой на устойчивые парсеры источников;
- добавить расписание обновления;
- хранить историю в базе данных;
- добавить контроль качества данных;
- добавить авторизацию dashboard;
- настроить мониторинг ошибок загрузки и расчета.

## 6. Итог

Проект закрывает ключевую задачу ТЗ: строит систему раннего предупреждения стресса рублевой ликвидности на публичных сигналах, агрегирует их в интерпретируемый LSI, показывает dashboard, проводит backtest и дает аналитический комментарий по состоянию рынка.
